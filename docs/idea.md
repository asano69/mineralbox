## mineral(鉱物)かore(鉱石)か
- 国際鉱物学連合では鉱物（mineral substance）を「地球や地球外の天体で、地質作用を経て自然に生成した固体」と定義しており、この粒子（鉱物）の集合体を岩石という。そして、この鉱物や鉱物の集合体（岩石）のうち、人の生活上役に立つもの、特に資源として有用なものを鉱石 ore という。
- 比喩的には、個々のコードスニペットファイルが mineral で、それらをまとめた単位が ore に相当すると思う。しかし、博物館やコレクターが集める美しい結晶標本や珍しい鉱物標本は、必ずしも ore とは呼ばない。ore は経済的な価値を生み出すたんなる資源としてみられるようなものだ。(iron ore, copper ore, gold ore) ore に相当するものは、AIが生成したプログラムそのものかもしれない。
- するとやはり、プロジェクト名としては、mineral をつかうほうが自然ということになる。問題は、specimenという単語をテーブル名などでつかうべきかということ。

テーブル名の例
```
minerals
---------
1 | AES encryption
2 | Binary search
3 | Rate limiting


specimens
----------
1 | mineral_id=1 | Python | cryptography.AES implementation
2 | mineral_id=1 | Go     | crypto/aes wrapper
3 | mineral_id=2 | Rust   | binary_search()
```
- mineralは、そのコードで実装されている機能の比喩

```
鉱物種 (mineral)
    └── 標本 (specimen)
          └── 個体差・産地・状態などの情報

機能・概念 (mineral)
    └── コードスニペット (specimen)
          └── 言語・実装・作者・日時などの情報
```

## ❌️ Gitリポジトリの特定コミットの標本化
mineralboxの目的に照らして不必要な機能と判断。
- そもそも、新しいミネラルを追加するのにGitの特定コミットハッシュと関連付けなければならないのは面倒くさすぎる。
- ソースコードファイルを一括インポートしても結局ひとつひとつ注意深く観察することになる。理解していないファイルを追加するべきではないという観点でも自動追加機能は不適切。

---

- リポジトリのプロジェクトの構造を保ったままノートをつくれるとよい。例えば、ソースコードがそのままノートブックになると便利だ。あるプロジェクトのコミットハッシュを指定すると、そのコミット時の状態をそのまま標本化できると便利だ。

ファイルサイズの問題  
- 例えばgo-giteaの場合、ソースコードは.gitなしで47MBある。もし.gitを含めたら400MBある。47MBはあまりに大きい。
- ユーザがノートを書いたソースコードのみDBにコピーするようにしないと、DBがすぐに1GBを超えるだろう。しかもその95%は使っていないみたいな状態になる。
- 有効なgitリポジトリのURLとコミットハッシュがあれば、いつでも必要なデータは取得できるはずだ。すべてのスナップショットをコピーする必要はない。

Gtieaの場合
- Giteaの場合はソースコードはデータベースに保存せず、gitリポジトリをファイルビューワのように表示させているだけだ。
- Giteaの場合は、ただソースコードを表示するだけではなく、ソースコードの編集機能はもちろんREADME.mdのレンダリング機能が統合されている。これは考えると興味深い。うまく使えば、今開発しようとしているアプリは不要なのではという気すらしてくる。
- しかし、Giteaにはソースコードとその注釈メモを横に並べてまとめて表示する機能はない。

ユーザはテンプレートリポジトリを作り込むことによってソフトウェア開発を学べると素敵だ。

プロジェクト構造はたんなる階層型のビューにすぎない。ノートブックのようなものだ。

ソースコードの隣にマークダウンでノートをとれるとよい。行番号を指定できたらさらに便利だ。




いかなるソフトウェアにおいても、プログラミング言語が同じならば、ある機能が実装されたコードが保存されいているファイル名はある程度決まっている。

ファイル名ごとにサンプルコードを集めて帰納的に実装方法を学ぶというやり方も実践的だ。

さらに一歩すすめて、テンプレートリポジトリにフォーカスするのもよい。

しかし、優れたテンプレートリポジトリをつくるためにはどこかにサンプルを収集するための箱が必要だ。

テンプレートリポジトリを作り込むことは魅力的だが、それよりも、テンプレートリポジトリを丸暗記してしまうことのほうが愉快だろう。

重要な前提
- Deepwikiをつかえばあらゆるソフトウェアの仕組みを詳細に観察できる。
- 浅い理解はほとんど意味がなく、重要なのは深く理解することだ。

優れた設計のソフトウェアは、AIをつかって徹底的に解析し、そのメモをつくろう。このアプリはそのメモを保存するためのもの。
